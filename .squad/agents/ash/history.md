- **Project:** Dave Ball arcade game — bouncing balls in a box, user draws boundary lines to claim territory, fill 80% to win
- **Stack:** Python, Docker, Web (HTML5 Canvas), WebSocket
- **Created:** 2026-03-26

## Learnings

<!-- Append new learnings below. Each entry is something lasting about the project. -->
- **2026-03-26 — Phase 2: physics + game_state implemented.** Created `backend/physics.py` (Ball class, `update_ball_position`, `check_wall_collision` with axis-aligned elastic reflection, `check_line_ball_collision` with point-to-segment distance + AABB pre-check) and `backend/game_state.py` (GameState class managing play area, balls, boundaries, growing lines, score/lives/state). All 12 `test_physics.py` tests pass. Added `backend/__init__.py` and `pythonpath = ["."]` to pyproject.toml for package imports. Updated `app.py` with 30Hz game loop via `socketio.start_background_task`, plus `start_game` and `line_start` socket events.
- **Import pattern:** `game_state.py` uses try/except for relative vs absolute imports to work both as `backend.game_state` (tests) and `game_state` (Docker direct execution). `physics.py` is self-contained with no internal backend imports.
- **2026-03-26 — Phase 4: Line Growth Mechanics implemented.** Added `GrowingLine` class to `game_state.py` with two-arm growth (arm1 = negative direction, arm2 = positive direction), boundary-aware target finding, and configurable `LINE_GROWTH_SPEED` (200px/s) + `MAX_GROWING_LINES=1` in `config.py`. `start_line()` validates play area, direction, boundary overlap, and one-at-a-time limit. `grow_lines(dt)` extends arms and auto-completes. `complete_line()` converts to permanent boundary + sets `territory_recalc_needed` flag. `fail_line()` decrements lives and triggers game-over at 0. Updated `app.py` game loop: balls bounce off completed boundaries, lines grow per tick, line-ball collision checked against arm segments. 27 new tests in `test_line_growth.py`, all 39 tests passing. `test_territory.py` still errors on import (Phase 5 module not yet built — expected).
- **2026-03-26 — Phase 5: Territory Calculation & Filling implemented.** Created `backend/territory.py` with Grid class (2D cell array, EMPTY/WALL/FILLED states, Bresenham line rasterization, coordinate clamping for edge boundaries), Region dataclass (cells set, is_filled flag, cell_count property), BFS `flood_fill()` (deque-based, stops at walls/grid edges), and `find_enclosed_regions()` (scans all non-wall cells, flood-fills contiguous regions, marks ball-free regions as filled). Ball-on-wall edge case handled by checking adjacent cells. Added `recalculate_territory()` to GameState: scales boundaries/balls to grid coords via GRID_CELL_SIZE, rasterizes, finds regions, computes fill_percentage, stores filled_regions as bounding boxes for rendering, triggers 'won' state at ≥80%. Wired into app.py game loop: territory recalc fires after line_completed, emits 'region_filled'/'game_won'/'game_lost' events. Fixed Lambert's test_fill_percentage_known_area assertion (wall at x=5 in 10-wide grid → right side is 4 columns=40 cells, not 5=50). All 50 tests passing (12 physics + 27 line growth + 11 territory).
- **2026-03-26 — Level Progression + Production Docker.** Added `next_level()` to GameState: increments level, ball count = level+1, speed scales by 1.1^(level-1), resets play area (boundaries/lines/fill) while keeping cumulative score, resets lives to 3 (INITIAL_LIVES). Ball spawning now uses retry loop to ensure minimum distance (4×radius) between balls and margin from edges. Added `next_level` socket event in app.py (only fires from 'won' state, restarts game loop). Created `backend/Dockerfile.prod` (gunicorn + eventlet worker), `frontend/Dockerfile.prod` (gzip-enabled nginx), `docker-compose.prod.yml` (no volume mounts, healthchecks, `service_healthy` dependency). Health endpoint `/health` was already present. All 50 tests still pass.
- **Ball Speed Setting.** Added `speed_multiplier` support: `config.py` has `SPEED_MULTIPLIER_OPTIONS = {slow: 0.5, normal: 1.0, fast: 1.5}`. `GameState.__init__` accepts `speed_multiplier` param (default 1.0), multiplies `BALL_SPEED` by it for initial ball speeds. `next_level()` also applies the multiplier alongside the per-level 1.1× scaling. `to_dict()` includes `speed_multiplier` for frontend display. `app.py` `start_game` handler parses optional `speed_multiplier` from client data; `next_level` handler can also update it mid-game. All 76 tests pass.
- **Dynamic Lives = num_balls + 1.** Changed lives system so each level gives lives = number of balls + 1 (L1: 2 balls → 3 lives, L2: 3 balls → 4 lives, etc.) instead of the old hardcoded `INITIAL_LIVES`. Updated `__init__()`, `reset()`, and `next_level()` in `game_state.py`. `INITIAL_LIVES` config constant is no longer used by GameState. All 76 tests pass — existing tests used relative assertions (`initial_lives - 1`) so no test changes needed.
- **Ball-to-Ball Collision + Fission.** Added elastic 2D ball-ball collision to `physics.py` (`check_ball_ball_collision`, `resolve_ball_ball_collision` — swaps velocity normal components, preserves tangent, separates overlapping balls). Added `check_ball_collisions()` to `GameState` (nested pair loop, resolves collisions, spawns one fission ball per collision at midpoint with random direction and average parent speed, capped at `MAX_BALLS=20`). `ball_collision_events` list tracks per-tick collision points; `to_dict()` includes `ball_collisions` field for frontend VFX. Wired into `app.py` game loop step 2 (after position update, before line growth). Lives NOT updated mid-level for fission balls. All 76 tests pass.
- **Fire Power-Up.** Added `"fire": 12` to `POWERUP_WEIGHTS` in `config.py`. Implemented fire power-up: capturing it sets `fire_active = True` on GameState (simple boolean like shield). On `start_line()`, if `fire_active` is True the `GrowingLine` gets `is_fire = True` and `fire_active` is consumed. Fire lines destroy balls on contact instead of failing — handled in both app.py mid-growth collision check (step 4) and game_state.py `grow_lines()` completion check. Destroyed balls are tracked in `fire_destroy_events` list (following `snake_eat_events` pattern). `GrowingLine.__slots__` updated to include `is_fire`; `to_dict()` includes `is_fire`. GameState `to_dict()` emits `fire_active` and `fire_destroy_events`. Events cleared after broadcast in app.py. All 234 tests pass.
- **Acid Power-Up.** Added `AcidPool` class to `game_state.py` (x, y, radius, timer, active flag + `to_dict()`). Config: `ACID_DURATION=12`, `ACID_POOL_RADIUS=35`, `ACID_POOL_COUNT_MIN=3`, `ACID_POOL_COUNT_MAX=5`, weight 10. `_apply_acid_effect()` spawns 3–5 pools at random unfilled positions using `_is_point_in_filled()`. `update_acid_pools(dt)` decrements timers, removes expired pools, checks ball-pool distance, dissolves balls (keeps ≥1 alive via `len(self.balls) - len(destroyed) > 1` guard). `acid_dissolve_events` follows `snake_eat_events` pattern. State cleared in `reset()` and `next_level()`. Called from `app.py` game loop alongside other entity updates. Frontend (Dallas) had rendering pre-wired — I added `ACID_COLORS`, acid icon/aura in `drawPowerUps`, and entries in `colorMap`, `textMap`, `textColorMap`, `resolvedMap`, `MYSTERY_SLOT_EMOJIS`, `MYSTERY_RESOLVED_EMOJI`. All 234 tests pass.
